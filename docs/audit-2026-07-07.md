# MeetMe audit — 2026-07-07 (4-agent Opus pass, consolidated)

Four independent Opus 4.8 agents audited: money/state-machine correctness, app UX
friction, edge-cases/dead-ends, and best-flow proposals. Findings below are
de-duplicated and cross-checked; the two headline money bugs and the security hole
were re-verified against the code by hand. `[V]` = verified against source.

Convergence signal: the UX and best-flow agents independently landed on the same
friction wins — strong confidence those are the right simplifications.

## Tier 0 — Money-movement bugs (fix before any real rail; #1 is a bug today)

The core double-entry ledger is **sound** — every outcome conserves, the $150 example
matches to the cent, the "buyer always gets ≥$1 deposit back" cap holds. The bugs are
all in the **rail-disbursement layer**, where accounting and real money can diverge.

1. **[V] Self-service dispute never disburses.** `MONEY_OUT` (`payments.ts:10`) omits
   `PROPOSE_RESOLUTION`. Both parties agree → `DISPUTE_RESOLVED`, ledger splits escrow
   back — but the payout/refund mirror loop (`payments.ts:57`) never runs → no rail
   transfer. Funds stuck. Uncaught because the test only asserts state.
   **Fix:** add `PROPOSE_RESOLUTION` to `MONEY_OUT`; better, key disbursement on
   whether `res.ledger` has outgoing bank credits, not on the action type.
2. **[V] Dispute-`release` skips the settlement gate.** `RELEASE` (`payments.ts:9`)
   excludes `RESOLVE_DISPUTE`/`PROPOSE_RESOLUTION`, so a dispute resolved to `release`
   pays the seller even if the buyer's ACH is pending/returned. Violates the
   "never pay before settle" invariant. **Fix:** gate on release *semantics*
   (ledger credits `bank:<seller>`), not the action type.
3. **No-show $5 compensation mislabeled as a refund.** The stood-up buyer's $5
   (captured from the seller's card) is emitted as a `refund_buyer` →
   `initiateRefund` (`payments.ts:65`). On a real rail that refunds against the
   buyer's original charge, so the extra $5 fails/over-refunds — **the buyer never
   receives the $5** the rules promise. FakeRail masks it. **Fix:** route captured
   compensation as a payout/transfer to the buyer, not a refund.

## Tier 1 — Security & frozen-funds (before any deploy)

4. **[V] `allowDev:true` hardcoded + unauthenticated `/dev/*` routes** (`index.ts`,
   `server.ts`). Anyone can `Bearer dev:<victimId>` to impersonate any user, and
   `/dev/deals/:id/{resolve,settle-funding,return-funding}` take no auth.
   **Fix:** env-gate (`ALLOW_DEV==='1'`), default OFF; never expose in a reachable env.
5. **Either party can resolve their own dispute in their favor.** The "specialist
   decides" Release/Refund/Split buttons render to *both* participants and call the
   unauthenticated dev resolve endpoint. **Fix:** remove from the user UI; real
   authenticated admin/staff path only.
6. **Dispute deadlock.** If the two never propose the same outcome, `DISPUTED` has no
   timeout and the worker never touches it → escrow frozen indefinitely. **Fix:**
   worker SLA (auto-escalate/auto-refund after N days) + real admin resolution.
7. **Chat has no server-side state gate** (`server.ts:326`). Messages can be sent on
   `DRAFT` and on terminal deals (post-deal harassment channel). **Fix:** gate sends
   to `AGREED`..non-terminal server-side (mirror the client's allowed set).

## Tier 2 — Latent correctness, safety & privacy

8. **FUND is non-atomic.** Ledger commits `ARMED`+funded before the rail pull; if
   `initiateFunding` throws after, no funding transfer exists → release gate blocks
   forever (bricked deal). **Fix:** record a pending transfer in the same commit.
9. **Card-hold written outside the version lock** — a `HEAD_OUT`/`CANCEL` race can
   stamp a hold onto an already-terminal deal → a live $5 auth never released.
   **Fix:** fold hold place/clear into the atomic commit or add a reconciliation sweep.
10. **`EXPIRED_NO_SHOW` reachable from `ARMED`** (`states.ts`) — captures the seller's
    $5 for a meetup that never started. Latent (worker only fires from EN_ROUTE).
    **Fix:** drop it from `ARMED`'s allowed transitions.
11. **Refunds aren't settlement-gated** — refunding an unsettled ACH that later returns
    = double loss. **Fix:** cancel the original transfer instead of forward-refunding.
12. **"Leave safely" silently forfeits the fleeing user's $5 + trust.** The panic exit
    fires `CANCEL`, which from EN_ROUTE is a self-declared no-show. **Fix:** waive the
    forfeit for safety exits (or clearly warn before executing).
13. **Counterparty's "coming from" (often home) pin leaks on the map** once EN_ROUTE if
    they don't re-share live GPS. Violates the "never show raw coordinates" invariant.
    **Fix:** only render live EN_ROUTE/AT_MEETUP pings, or an approximate marker.
14. **Trust penalties erased by any later rating** — `trust_score` is recomputed from
    the rating average, overwriting no-show/at-fault deltas. **Fix:** blend
    rating-average minus a penalty accumulator.
15. **Liveness dead-ends:** EN_ROUTE with neither arrived, and AT_MEETUP with a stalled
    code exchange, have no worker timeout → funds frozen. **Fix:** add both timeouts.
16. **Can't set a meetup spot after heading out** — `SET_MEETUP` rejected past ARMED
    and the UI hides it. **Fix:** allow through EN_ROUTE.
17. **Accept-invite race → duplicate deals** (no conditional update) and accept isn't
    bound to the invited phone. **Fix:** conditional `where status='pending'` + phone check.
18. **Demo direct-create doesn't handle `kyc_required`** (invite path does). **Fix:** mirror it.
19. **No push on `CANCELLED` or `DISPUTED`** — counterparty not told. **Fix:** add messages.

## Tier 3 — Frictionless-flow wins (the "best flows" — ~20 taps → ~7)

Both UX agents converged: **the friction is the app asking users to do things the
system already knows how to do.** The backend already supports geofence auto-arrival,
midpoint spot ranking, and instant settlement — the client just isn't using it.

- **A. Empty out the seeded fake phone numbers** on the real login + invite fields
  (`AppContext.tsx`). One-line fix; worst first-run confusion.
- **B. Merge Accept + Terms + Fund → one "Accept & Fund $155".** Capture the seller's
  card/terms at invite; land invite-accept in `AGREED` (not DRAFT); fold the buyer's
  accept into fund. Kills the "why am I agreeing to my own deal?" double-accept and a
  dead-end where the invitee waits on the inviter.
- **C. Auto-suggest the #1 safe meetup spot, one-tap Confirm** (server already ranks by
  balanced drive-time). Replace the open→choose-source→wait-for-both→browse modal with
  a proposed default + a "Change" escape hatch.
- **D. "Heading out" auto-starts live location; arrival auto-detects** via the existing
  60m geofence (`location.ts` `CO_LOCATED`). Drop the manual "Share location" and
  "I've arrived" buttons; make the map truly live (Uber-style).
- **E. Collapse the code dance** — auto-reveal on AT_MEETUP + auto-submit on the 4th
  digit; consider QR scan. *Tradeoff:* dropping `CONFIRM_RECEIVED`/`CONFIRMING` removes
  a dispute window — keep one buyer "Release" consent tap as the guard.
- **F. Shareable invite deep-link** — solves cold-start ("drop a link in the marketplace
  chat") and kills the phone-typing + role toggle + "Text them?" prompt.
- **G. Quick wins:** OTP autofill (`textContentType="oneTimeCode"`) + strip the
  "(local: 123456)" hint + `loading` on auth buttons; remember last invite role;
  contacts picker; plain-dollar amount entry (or helper text); move logout + theme
  toggle to Account only; replace the raw ledger jargon with `WalletSplit`;
  consolidate the four stacked danger buttons into one tiered "Get help / leave".
